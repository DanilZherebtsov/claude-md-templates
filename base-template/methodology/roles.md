# Роли — специализированные чаты-сотрудники

Роль — профиль чата для одной функции (примеры — <<СЛОТ ROLES-FILL: роли класса, напр.: юрист, финансист, методолог, security-reviewer>>): «нанять» функцию на задачу, не размывая соседними. Определения — в `roles/<роль>.md`; образец — [roles/_шаблон.md](../roles/_шаблон.md).

## Создание роли — «создай роль <название>»

По фразе «создай роль X» Claude: задаёт 2–4 наводящих вопроса (зона ответственности, границы, срез вики — префиксы/теги, куда писать находки), копирует [roles/_шаблон.md](../roles/_шаблон.md) в `roles/<роль>.md`, заполняет и объявляет срез. **CLAUDE.md менять не нужно:** механизм ролей в нём уже описан, новая роль подхватывается по своему файлу, список ролей — `ls roles/`.

## Вызов

«прочитай `roles/<роль>.md` и работай в этой роли», дальше — задача. Claude читает файл роли, `CLAUDE.md` и стартовые страницы вики из роли, затем работает строго в зоне роли (выходит — только чтобы перенаправить).

## База знаний роли — срез общей вики, а не отдельная вика

У роли нет своего хранилища. Её «база знаний» — **срез одной общей вики** (по префиксам/тегам) плюс связанные сырые файлы. Почему не отдельная вика: решения опираются на знание из разных функций сразу — оно должно лежать в одном графе; отдельная вика на функцию рвёт связи и плодит дубли.

## Ретрив и сохранение

- **Ретрив (не вчитывая всю вику):** срез задан префиксами/тегами роли. Claude находит страницы по `wiki/index.md` и префиксу имени (`ls <type>/<префикс>-*`), при необходимости — `grep` по `tags:`; читает только совпавшие, на шаг по ссылкам.
- **Сохранение:** новые находки — в общую вику обычным [ingest](ingest.md) (с `sources:`, человек подтверждает); черновой разбор — в рабочий слой класса; решение — ADR в `decisions/`.

## Правила

- Роль не пишет в вики молча: через [ingest](ingest.md) с источником.
- Роль не решает за владельца: готовит рекомендацию; выбор и ADR — через lifecycle-файл класса.
- Роли заводит и правит человек (образец `_шаблон.md`); это рабочая область, не методология.
