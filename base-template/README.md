# base-template — общий скелет шаблонов

Это **не шаблон для целевого проекта**, а скелет, из которого собираются шаблоны классов (saas-product, research-project, business-run-general и будущие). Здесь живёт **type-независимое ядро** LLM-вики: операции, дисциплина, механика STATE, роли, bootstrap. Что специфично для класса — помечено `<<СЛОТ: …>>` и заполняется при деривации.

**Отношение к корневому `CLAUDE.md` мета-проекта:** корневой CLAUDE.md — это методология *как мы строим шаблоны* (правила самого мета-проекта). `base-template` — переиспользуемый *скелет шаблона* для целевого проекта. Корневой объявляет механику вики инвариантом; base делает её исполняемой, чтобы не переизобретать в каждом классе. Решение — [ADR-0008](../../wiki/decisions/adr-0008-base-template.md).

## Как собрать шаблон класса из base

1. Скопировать `base-template/` в `templates/<класс>/`.
2. Заполнить все `<<СЛОТ: …>>` (грепом `grep -rn "<<СЛОТ"` — найти, ни одного не оставить).
3. Добавить доменный lifecycle-файл в `methodology/` (слот S6).
4. Прогнать глазами CLAUDE.md: убрать неподходящие дефолтные типы вики, дописать доменные.
5. Записать в мета-вику: ADR на новый класс + страница класса в `wiki/discovery/`.

## Список слотов (граница base ↔ класс)

| Слот | Что заполнить | Примеры по классам |
|---|---|---|
| **S1 «О проекте»** | 1–2 строки про дело + ограничения | saas: что строим + тех.выборы · research: главный вопрос + аудитория · business: что за дело + бюджет/юр.форма |
| **S2 набор type-папок `wiki/`** | какие типы у класса (база даёт `decisions`/`discovery`/`synthesis`/`principles`; убрать лишнее, добавить своё) | saas: +`architecture/` · research: `claims/` вместо `discovery/` · business: +`entities/` |
| **S3 центральный/доменный тип + его формат** | описать формат ключевого типа в page-conventions; **при необходимости — свою lint-проверку** (`methodology/lint.md`, слот S3-lint) | saas: architecture (+`implementation:` + path-check в lint) · research: claim-evidence chain (+ локализация цитат в lint) · business: entity-карточка |
| **S4 доп. рабочие слои** | `output/` есть у ВСЕХ классов (фиксированный корень рабочих файлов; внутри — любая структура по факту). Слот = **дополнительные** слои для классов с кодом | saas: +`specs/`+`src/`+`data/` · research: +`data/`+`src/`(опц) · business: только `output/` |
| **S5 разделы STATE** | фиксированный список разделов | saas/research: стадия/путь/в работе/следующее/завершено/блокеры/(7-й) · business: профиль/направления/неделя/следующее/обязательства/блокеры/завершено |
| **S6 доменный lifecycle-файл** | поток единицы работы | saas: `spec-lifecycle.md` · research: `question-lifecycle.md` · business: `decision-lifecycle.md` |
| **S7 правило авторитета** | кто побеждает при конфликте | классы с кодом: «код побеждает вики» · без кода: «источники побеждают вики» (+ research: «цитата с локализацией») |
| **S8 доменные конвенции** | валюта, единицы, спец-форматы | business: рубли по умолчанию · research: цитаты `(Автор год, с.)` |

### Слоты в `methodology/` (ADR-0013)

Доменная вариация размечена слотами не только в CLAUDE.md, но и в methodology-файлах:

| Слот | Файл | Что заполнить |
|---|---|---|
| **KNOWLEDGE-UNIT** | ingest | единица компиляции: страница (дефолт) / `claim` (research) |
| **RAW-DEFAULTS** | bootstrap | стандартные подпапки `raw/` класса |
| **INTERVIEW-Q** | bootstrap | вопросы интервью класса |
| **DOMAIN-LINT** | lint | проверка центрального типа (code-drift / claim-баланс / календарь) |
| **CLOSE-OP** | index-log-format | доменная операция закрытия в логе |
| **ROLES-FILL** | roles | примеры ролей класса |

Полная карта СТАНДАРТ vs КАСТОМ + правила кастомизации — [methodology/template-anatomy.md](../../methodology/template-anatomy.md).

### Про `raw/`

base по умолчанию делает подпапки `raw/` динамическими (создаёт bootstrap, как в business). Но у классов со **стандартным набором источников** (research: `literature/`, `fieldwork/`; saas: `discovery/`, `decisions/`, `technical/`) — оставьте доменный стартовый набор подпапок в дереве CLAUDE.md + правило роста; полностью «с нуля» — только для классов без устойчивого набора.

Всё, что не слот, — общее ядро: правится **только здесь**, потом раскатывается в производные (см. ниже).

## Сопровождение

Правка общей механики (СТАНДАРТ — каркасы ingest/query/lint, дисциплина, фаза «подумай», bootstrap, роли, state-механика) идёт **в base один раз**, затем раскатывается операцией «распространить base» ([methodology/propagate.md](../../methodology/propagate.md)) с подъёмом версий ([ADR-0011](../../wiki/decisions/adr-0011-versionirovanie-i-rasprostranenie-shablonov.md)). Граница СТАНДАРТ/КАСТОМ и правило **«лифт, не форк»** (улучшил generic-часть в классе — подними в base, не оставляй в классе) — [ADR-0013](../../wiki/decisions/adr-0013-standart-vs-kastom-granica-shablona.md) / [methodology/template-anatomy.md](../../methodology/template-anatomy.md).
