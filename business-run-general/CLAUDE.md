# CLAUDE.md — LLM-вики для ведения бизнеса и личных проектов

> **Шаблон:** business-run-general · v4. На lint Claude сверяет версию с апстримом шаблона и предлагает обновления — [methodology/lint.md](methodology/lint.md).

Базовая идея: не выводить одно и то же знание из сырых материалов заново при каждом вопросе. Вместо этого — однократно компилировать каждый источник (договор, разговор, прайс, ресёрч, КП) в постоянную, взаимосвязанную вики. Дальше Claude читает вики, а не сырые материалы, когда отвечает. К сырому возвращается только тогда, когда нужно интегрировать новые данные или разрешить противоречие.

Корневой файл содержит **always-on правила и указатели**. Детальные процедуры (ingest, query, lint, конвенции страниц, lifecycle решений, роли, bootstrap) — в [methodology/](methodology/), читаются по триггеру.

## Архитектура

Структура в корне. Центральные слои — `raw/` (входные материалы) и `wiki/` (скомпилированные знания). Рядом — методология, роли и рабочая поверхность для результатов:

```
raw/             ← Сырые источники. Вы добавляете, Claude только читает.
                   Подпапки заводит bootstrap под ваши источники и вписывает их
                   сюда; дальше расширяются по мере надобности (глубина свободна).
wiki/            ← Скомпилированные знания. Управляет Claude, поддерживается автоматически.
                   Пять type-папок, плоско (никаких подпапок внутри типа):
  decisions/     (решения — что выбрали и почему; ADR, связаны через supersession)
  entities/      (контрагенты и люди: поставщики, клиенты, подрядчики, сотрудники,
                  арендодатели, партнёры. Одна страница на сущность, история копится.
                  Группировка префиксами: supplier- client- contractor- staff- landlord- partner-)
  discovery/     (знание о внешнем мире: рынок, конкуренты, цены, нормативы,
                  юридические и налоговые факты. Префиксы: market- competitor- norm-)
  synthesis/     (записанные обратно ответы: сравнения, ретроспективы, брифинги)
  principles/    (уроки «больше так не делаем» и рабочие правила, выведенные из опыта)
  index.md       (каталог — одна строка на страницу)
  log.md         (лог операций, только дополнение)
methodology/     ← Инструкции для Claude: процедуры операций и форматы.
                   Корневой CLAUDE.md содержит указатели; детали — здесь.
                   Часть шаблона, читается по триггеру.
  ingest.md            (процедура добавления знания)
  query.md             (процедура структурированного ответа)
  lint.md              (правила проверки целостности)
  page-conventions.md  (frontmatter, типы страниц, формат)
  decision-lifecycle.md (поток решения: вопрос в STATE → сравнение → ADR)
  roles.md             (как работают специализированные чаты-роли)
  state-rules.md       (структура и правила обновления STATE.md)
  index-log-format.md  (формат index.md и log.md)
  bootstrap.md         (как развернуть структуру с нуля)
roles/           ← Определения специализированных чатов-ролей (юрист, финансист,
                   маркетолог…). Авторские, заводятся под ваше дело. `_шаблон.md` — образец.
output/          ← Рабочие файлы. Корень создаётся пустым; подпапки заводятся
                   по смыслу: `черновики/` (расходное) + папки под живые документы.
CLAUDE.md        ← Этот файл.
STATE.md         ← Операционное состояние: профиль момента, открытые направления,
                   календарь обязательств. Не canonical (намерения, не факты).
```

Правила:

- **`raw/` неизменяем.** Только добавление. Никогда не редактировать и не удалять — оригинальная формулировка важна, когда мы потом перепроверяем цифру или утверждение.
- **`wiki/` под управлением Claude.** Человек читает, но не правит руками — все мутации wiki делает Claude. Растёт через несколько каналов: (а) ingest сырого источника из `raw/`; (б) запись-обратно содержательного ответа в `synthesis/` после query; (в) извлечение ADR из принятого решения; (г) запись принципа из инцидента или разговора. Решения и заметки, возникшие в чате, попадают в вики тем же путём: Claude сохраняет запись разговора в подходящую подпапку `raw/`, дальше — ingest. Если в вики что-то неверно — исправьте источник в `raw/` (или укажите Claude, где ошибка), и Claude перекомпилирует.
- **Глубина wiki = 1.** Внутри `wiki/` — только один уровень тематических подпапок (пять типов выше), глубже не идти. Если в одном type-folder скапливается 30+ однородных страниц — это сигнал к **горизонтальному расширению** (новый top-level type или плоская группировка через префиксы имён), не к подпапкам. См. «Дисциплину», правило про расширения схемы.
- **`raw/` — исключение.** Внутри `raw/` глубина по необходимости разрешена: это хранилище под навигацию человеком (например, подпапка на крупного контрагента или проект).
- **`methodology/` — часть шаблона, не рабочая область.** Правки только человеком и только при пересмотре методологии. Расширение разрешено (новые файлы для новых операций); переписывание существующих требует осознанного решения.
- **Источники побеждают вики.** Если страница в `wiki/` говорит одно, а оригинальный источник в `raw/` — другое: истина в источнике, пока новый ingest или ADR не зафиксирует обратное. Вики — наш скомпилированный конспект, не первоисточник.
- **git опционален.** Если проект под версионным контролем — Claude коммитит после ingest/bootstrap (см. методологию). Если нет (типично для работы в Cowork) — история живёт в `wiki/log.md` и в датированных записях самих страниц; шаги «закоммить» просто пропускаются.
- **Структура растёт по мере надобности.** Не подошёл источник под существующие подпапки `raw/` — заведи новую (в `raw/` глубина свободна; в `wiki/` — плоско, глубина = 1). Изменил раскладку — отрази её в дереве этого файла и в затронутых контекстных файлах.
- **Имена файлов.** На русском, через подчёркивание, описательные. Сырые файлы префиксуются датой `YYYY-MM-DD-`, если известна дата источника.

## Иерархия источников

В одной сессии Claude видит несколько источников знаний. Чтобы не было путаницы, какой первичный, — порядок жёсткий:

1. **CLAUDE.md** (этот файл) — always-on правила и контекст.
2. **`methodology/`** — детали операций и форматов. Читаются по триггеру, не в фоне.
3. **Вики** (`wiki/`) — canonical-знание, накопленное через ingest.
4. **Сырые источники** (`raw/`) — первоисточник, к которому возвращаемся при сомнениях.
5. **Auto-memory Claude** (`MEMORY.md`, `memory/`) — быстрый кэш между сессиями, **не canonical**.

**Конфликт между memory и вики.** Всегда верить вики, не memory. Memory может быть устаревшей; вика отражает последний ingest.

**Конфликт между вики и сырым источником.** При расхождении — перепроверить источник и исправить вики. Не предполагать, что вики «знает лучше» — именно так заводится дрейф.

**Запись в memory.** Когда появилось новое знание о деле — Claude сначала кладёт его в `wiki/` через ingest. В memory, опционально, оставляет **короткий указатель**: тип записи + ссылка на страницу вики. Длинное содержание не дублировать.

**`STATE.md` — отдельная ось, не часть иерархии выше.** STATE описывает текущие намерения и обязательства, а не утверждения о мире. С wiki не конфликтует (разные домены: wiki — что мы знаем, STATE — что мы делаем и кому должны).

## Операционное состояние

`STATE.md` в корне — единственное место, где живут профиль момента, открытые направления и календарь обязательств. **Структура — фиксированный набор из 7 разделов** (профиль, открытые направления, на этой неделе, следующее, календарь обязательств, блокеры/риски/ждёт решения, недавно завершено). Пустые разделы остаются с пометкой `_пусто_`.

Детали — структура, правила обновления, триггеры свежести, связь с wiki, ведение календаря обязательств — в [methodology/state-rules.md](methodology/state-rules.md). Прочитайте перед первой работой со STATE или при ревизии правил.

**Триггеры чтения STATE.md (для Claude):**
- При старте сессии — молча читать STATE.md. Это даёт контекст «на чём остановились» без явного вопроса.
- На вопросы «на чём остановились», «что в работе», «что дальше», «какие обязательства/дедлайны», «что ждёт моего решения» — STATE.md основной источник ответа.
- Если поле `_Обновлено:_` старше 7 дней — в первой реплике сессии предложить «STATE устарел на N дней, что изменилось?».

**Все стартовые проверки молчаливы.** Свежесть STATE, свежесть lint (§5 «Дисциплины») и ближайшие/просроченные обязательства календаря упоминаются в первой реплике **только при отклонении**. Сказать нечего — Claude молчит, «всё в порядке» не перечисляет.

## Wiki: типы страниц и операции

Wiki содержит **5 типов страниц**:

- **`decisions/`** — решения (Architecture Decision Records). Что выбрали и почему: поставщик, локация, цена, найм, юр. форма. Решения помечаются как `superseded`, а не удаляются.
- **`entities/`** — контрагенты и люди. Одна страница на сущность (поставщик, клиент, подрядчик, сотрудник, арендодатель, партнёр); история отношений копится журналом. Группировка префиксами имени файла: `supplier-`, `client-`, `contractor-`, `staff-`, `landlord-`, `partner-`.
- **`discovery/`** — знание о внешнем мире: рынок, конкуренты, цены, нормативы, юридические и налоговые факты. У гипотез — явный статус (`open`, `validated`, `invalidated`).
- **`synthesis/`** — записанные обратно ответы: сравнения, ретроспективы, сквозные анализы, брифинги.
- **`principles/`** — уроки «больше так не делаем» и рабочие правила, выведенные из инцидентов и повторяющихся паттернов. Читаются Claude перед нетривиальной задачей.

Frontmatter, формат шаблонов по типам, страницы-журналы, перекрёстные ссылки — [methodology/page-conventions.md](methodology/page-conventions.md).

**Три операции** работают с wiki. Запускаются обычной фразой в чате; Claude сам определяет по смыслу, какую вы имеете в виду.

- **Ingest** («обработай этот документ», «запомни это», «добавь в вики», «зафиксируй») → [methodology/ingest.md](methodology/ingest.md).
- **Query** («что мы знаем про X», «сделай справку по Y», «сравни A и B») → [methodology/query.md](methodology/query.md).
- **Lint** («прогони lint», «проверь вики») → [methodology/lint.md](methodology/lint.md).

**Поток решения** (открытый вопрос в STATE → сравнение в `output/черновики/` → ADR в `decisions/` при выборе → каскад в `entities/`/`principles/`) — [methodology/decision-lifecycle.md](methodology/decision-lifecycle.md).

**Формат `index.md` и `log.md`** — [methodology/index-log-format.md](methodology/index-log-format.md).

## Роли (специализированные чаты-сотрудники)

В папке `roles/` лежат описания ролей — например, `юрист.md`, `финансист.md`, `маркетолог.md`. Это правила работы для специализированных чатов. Если первое сообщение в чате просит «прочитай `roles/юрист.md` и работай в этой роли» — Claude следует описанию роли строго: работает только в её зоне ответственности, не выходит за неё, фиксирует находки в указанные в роли файлы вики. Механика и формат роли — [methodology/roles.md](methodology/roles.md). Завести роль — фразой «создай роль <название>»: Claude задаёт наводящие вопросы и создаёт файл роли со срезом (префиксы/теги) по той же [methodology/roles.md](methodology/roles.md).

## Дисциплина (то, что не даёт вики сгнить)

1. **Фильтровать на входе, а не на хранении.** Если источник не стоит интеграции — не интегрируй его. В вики попадает только то, что ты готов защищать.
2. **Supersession вместо тихого исчезновения.** Никакого автоматического забывания. Старый материал остаётся, помеченный как `superseded` и связанный с заменой. Решение, отменённое полгода назад, часто ценнее свежего — оно объясняет, почему появилось текущее.
3. **Никакой ложной точности.** Никаких числовых оценок уверенности. Достоверность утверждения видна из цепочки источников и связанных решений. «Два КП и один разговор» бьёт «0.85» каждый раз.
4. **Человек в петле записи.** Claude предлагает; человек подтверждает любую нетривиальную мутацию вики. Автоматический ingest соблазнителен, но он молча портит вики. Медленнее — это нормально.
5. **Lint не опционален.** Прогон раз в неделю. **Claude отслеживает свежесть:** при старте сессии читает дату последней записи `## [YYYY-MM-DD] lint |` в `wiki/log.md`; если прошло > 7 дней — предлагает прогон. Если отчёты lint растут быстрее, чем вы успеваете их разбирать — пересматривать нужно схему, а не вики.
6. **Сначала схема, потом механизм.** Когда что-то ощущается неправильно — сначала поправьте этот файл или соответствующий файл в `methodology/`, а не плодите обходные пути. Большинство «проблем тулинга» — это замаскированные проблемы схемы.
7. **Расширение схемы — только по горизонтали.** Допустимые расширения: новый top-level type в `wiki/` (плоско, рядом с пятью существующими) или новая top-level папка в корне. **Углублять имеющиеся типы запрещено.** Никаких `wiki/entities/suppliers/`, `wiki/decisions/2026/`. Если в одном type-folder скапливается 30+ однородных страниц — расширяйте горизонтально либо группируйте через префиксы имени. `raw/` — исключение.
8. **Синтез знаний при закрытии единицы работы.** Когда единица работы закрывается (решение принято, веха пройдена), Claude обязательно ревизует, какое новое знание она породила, и предлагает занести его в `wiki/` по всем релевантным типам (`decisions/`, `entities/`, `principles/`), не только профильному. Человек подтверждает (ADR/принципы — никогда молча). Пропуск = вики отстаёт от того, что мы реально знаем. Единицу работы задаёт [methodology/decision-lifecycle.md](methodology/decision-lifecycle.md).

## Принципы работы Claude в задачах

Эти правила применяются ко всем задачам — анализ, документы, переписка, решения. Они не про вики, а про то, как Claude действует.

Каждая нетривиальная задача проходит две фазы. Сначала **остановись и подумай**, потом действуй. Пропустить фазу «подумай» — самая частая причина того, что Claude уходит в ненужную работу или встраивает в неё ошибочные предпосылки.

**Нетривиальная задача** — та, где Claude должен **сделать выбор** (между подходами, контрагентами, формулировками) или **построить план** (несколько шагов или несколько файлов). Конкретные триггеры: затрагиваются деньги / юридические обязательства / отношения с людьми; нужно знание нескольких источников; есть риск исказить цифру или факт; непонятно, что точно нужно. **Тривиальные** (фаза «подумай» не нужна): опечатка, переименование, чтение и пересказ одной страницы. **В сомнении — считать задачу нетривиальной.**

### Перед началом — фаза «подумай»

**Закрой все запросы в сообщении (даже для тривиальных задач).** Если просьб несколько — перечисли их списком в начале ответа и закрой каждую; конкретные action-items (команды, пути) давай первыми, до анализа. Не зарывайся в один под-вопрос, теряя остальные — повторяющаяся ошибка.

1. **Перечитай релевантные принципы.** Если задача процессная — `wiki/principles/process.md`. Доменная — `wiki/principles/domain.md`. Принципы — это извлечённые из прошлых случаев правила; пропуск этого шага = повторение прошлых ошибок.
2. **Сформулируй предположения.** Назови вслух, что считаешь данным. Если есть неопределённость — спрашивай, не угадывай.
3. **Покажи разные трактовки.** Если запрос можно понять двумя способами — обозначь оба, не выбирай молча.
4. **Не выдумывай.** Цифры и факты — только из источников или вики, с указанием источника. Если данных не хватает — `[требует уточнения]` вместо угаданного значения.
5. **Один вариант за раз.** Не предлагай десять решений сразу. Обсуждай по одному, начиная с самого простого.
6. **Возражай, когда оправдано.** Если запрос ведёт к плохому решению (рискованная трата, юр. ловушка) — скажи об этом, а не выполняй молча.

### В процессе — фаза «действуй»

1. **Одна задача за раз.** Не перескакивать на соседние улучшения. Доводить текущее до конца.
2. **Сначала простота.** Минимальный текст или расчёт, который решает задачу. Никаких спекулятивных усложнений «на будущее».
3. **Точечные изменения.** Менять только то, что необходимо. Заметили несвязанную проблему — упомянуть, но не править.
4. **Выполнение от цели.** Для каждой нетривиальной задачи — сформулировать критерии успеха перед началом.

### После — фиксация принципа, если он появился

Если из задачи родилось правило вида «всегда делать X / никогда Y» — Claude предлагает записать его в `wiki/principles/<applicability>.md` с источником (ссылка на случай в `raw/` или на текущую дискуссию). Принципы записываются только из конкретных случаев, не из общих соображений.

## Документы и нейминг

Касается **рабочих результатов Claude** — КП, презентаций, отчётов, документов для согласования. Артефакты внутри `wiki/` живут по правилам [methodology/page-conventions.md](methodology/page-conventions.md).

- **Куда класть — три адреса.**
  - Прислали извне, не правим → `raw/` (неизменяемо).
  - Знание (факт, решение, контрагент) → `wiki/` через ingest.
  - Рабочие файлы → `output/`: расходное в `черновики/`; для живого документа, который ведёте, заведите папку по смыслу по мере появления (не заранее), прошлые редакции сохраняйте, не затирайте.
- **Имена файлов.** На русском, через подчёркивание, с датой когда уместно: `Сравнение_локаций_2026-05-25.md`.
- **Даты.** В тексте — `ДД.ММ.ГГГГ`. В именах файлов и YAML — `YYYY-MM-DD`.
- **Денежные суммы.** По умолчанию рубли с пробелом-разделителем тысяч: `220 000 ₽`. Другие валюты — по соглашению самой валюты (`$1,200`, `€990`). Зафиксируйте основную валюту в разделе «О проекте».
- **Цифры в документах.** Всегда с источником (файл, строка или ссылка). Без источника — пометка `[требует уточнения]` вместо угаданного значения.
