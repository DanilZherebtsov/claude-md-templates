# Конвенции wiki-страниц — frontmatter, типы, формат

Прочитайте этот файл:
- перед созданием новой wiki-страницы;
- перед редактированием frontmatter;
- при ревизии правил оформления.

## Frontmatter

Каждая страница wiki начинается с YAML-заголовка:

```yaml
---
type: decision | entity | discovery | synthesis | principles
status: active | superseded | draft | inactive | open | validated | invalidated
updated: YYYY-MM-DD
sources:                  # обязательно
  - ../../raw/договоры/2026-05-18-dogovor-arendy-lesnaya.md
  - ../../raw/переписка/2026-05-20-zvonok-s-postavshchikom.md
  - https://www.nalog.gov.ru/...
supersedes: [adr-0007]    # опционально
superseded_by: [adr-0014] # опционально
tags: [аренда, локации]
---
```

**Поле `sources:` обязательно.** Утверждение без источника — это галлюцинация, замаскированная под факт. Lint не пропускает страницы без источников. Источником может быть файл в `raw/`, внешний URL или явная запись разговора (сохранённая в `raw/`).

**Статусы.** `active` / `superseded` — для decisions, synthesis, discovery, principles. `draft` — для черновика любого типа. `inactive` — для `entity`, с которой больше не работаем (не удаляем — история ценна). `open` / `validated` / `invalidated` — для гипотез в `discovery/`.

## Атомарность

Одна концепция на страницу. Мягкий лимит — 400 строк, жёсткий — 800. Если страница перевалила за 400 — выделите подстраницы и свяжите их. Ограниченный размер означает, что любое отдельное чтение остаётся в пределах контекстного окна.

## Перекрёстные ссылки

Везде markdown-ссылки с относительными путями:
- `[Название](../type/slug.md)` для wiki-страниц;
- `[Описание источника](../../raw/концерн/file.md)` для сырых файлов;
- `[Название](https://...)` для внешних URL.

Анкор-текст обязателен и человекочитаемый — не slug файла. Единый формат обеспечивает рендеринг в любом просмотрщике (включая GitHub) и согласованность с полем `sources:` в frontmatter.

## Страницы-журналы

Если страница накапливает наблюдения во времени (типично для `entities/`, `discovery/`, `synthesis/`), каждая запись внутри начинается с `## [YYYY-MM-DD] Заголовок`. Это даёт возможность читать, что свежо, а что устарело, без необходимости разбирать историю изменений.

## Префиксы имён файлов

Внутри плоской type-папки страницы группируются **префиксом имени** — ведущим токеном файла (`supplier-kofe-import.md`, `norm-usn.md`, `client-acme.md`). Префикс заменяет подпапки (сохраняет глубину = 1) и служит ключом фильтрации: его видно в ссылках в `wiki/index.md` и через `ls discovery/norm-*`. Так роль или query находит свой срез, не вчитывая всю вику. Конкретные префиксы по типам — в разделах ниже.

## Типы страниц — формат

### decisions/ — решения (ADR)

Применяются к любым осмысленным выборам: поставщик, локация, цена, найм, юр. форма, отказ от варианта. Лёгкий формат:

```
# ADR-0012 — Выбрать локацию на Лесной для второй точки

Status: active
Date: 2026-05-18
Context: <проблема и ограничения, вынудившие сделать выбор — что выбирали, из чего>
Decision: <что решили, одним предложением>
Consequences: <что становится проще, что — сложнее; цифры с источником>
Alternatives considered: <кратко, с обоснованием отказа от каждого>
Sources: <ссылки на сравнение в synthesis/, КП, договоры>
```

Решения **помечаются как superseded, а не удаляются.** Старая запись остаётся читабельной, с заголовком, указывающим на её замену. Устаревший ADR всё ещё объясняет, почему появился текущий. Поток от вопроса до ADR — [decision-lifecycle.md](decision-lifecycle.md).

### entities/ — контрагенты и люди

Одна страница на сущность, с которой у дела есть отношения: поставщик, клиент, подрядчик, сотрудник, арендодатель, партнёр. Имя файла — по префиксу: `supplier-`, `client-`, `contractor-`, `staff-`, `landlord-`, `partner-`. Это сохраняет инвариант «глубина = 1» и позволяет фильтровать через `ls supplier-*`.

Формат тела:

```markdown
# <Имя> — <тип: поставщик / клиент / подрядчик / сотрудник / арендодатель>

**Status:** active | inactive
**Кто:** одна строка — чем занимается, как связан с нами.
**Контакты:** ответственное лицо, телефон, email (или `[требует уточнения]`).
**Условия:** ключевые коммерческие/договорные условия (цена, сроки, скидки, периодичность) — каждая цифра с источником.

## История
- [YYYY-MM-DD] Что произошло: первый контакт, прислали КП, подписан договор, спор, продление. Источник.

## Связи
- **Решения:** [ADR-0012 — выбрали как поставщика кофе](../decisions/adr-0012-postavshchik-kofe.md)
- **Документы:** [Договор поставки](../../raw/договоры/2026-05-18-postavka.md)
- **Обязательства:** см. «Календарь обязательств» в `STATE.md`, если есть повторяющийся платёж.
```

**Знание против операционки.** Стабильное (кто это, условия, история) — здесь. Текущий открытый вопрос с этим контрагентом (неоплаченный счёт, висящее решение) — в `STATE.md`, со ссылкой на эту страницу. Не дублировать.

### discovery/ — знание о внешнем мире

Одна страница на сегмент рынка, конкурента, ценовой ориентир, норматив, юридический или налоговый факт, содержательный инсайт. Имя файла — по префиксу: `market-`, `competitor-`, `norm-`, `price-`. У гипотез — явный статус (`open`, `validated`, `invalidated`) и доказательства, которые его сдвинули.

### synthesis/ — записанные обратно ответы

Страницы из содержательных запросов: сравнения (локаций, поставщиков, тарифов), ретроспективы, сквозные анализы, брифинги. Структура свободная, но обязательны: введение (что за вопрос, зачем синтез) и ссылки на источники/страницы вики, на которых синтез построен. Synthesis **переписываем**: при появлении нового знания мы скорее редактируем существующий, чем создаём новый.

### principles/ — уроки и правила

Когда из инцидента, повторяющейся ошибки или удавшегося паттерна выводится правило вида «всегда делать X / никогда Y» — оно записывается сюда, в страницу по applicability (`process.md`, `domain.md` — или другое разбиение по необходимости).

Внутри страницы каждая запись:

```markdown
## P-XX — Краткое имя принципа

**Правило:** что делать / что не делать.

**Почему:** контекст, инцидент или соображение, из-за которого правило возникло.

**Когда применять:** триггер (тип задачи, контекст).

**Прецедент:** дата + один-два кейса, где правило сработало или было нарушено.

**Источник:** ссылка на инцидент в `raw/` или связанный ADR.
```

**Триггер записи (для Claude):** когда из инцидента появляется правило, применимое к будущим задачам того же типа, — Claude предлагает: «это похоже на принцип P-XX в `wiki/principles/<applicability>.md`, записать?». Принципы — не теория из головы, а извлечение из конкретного случая; без источника принцип не записывается.

**Триггер чтения (для Claude):** перед началом нетривиальной задачи Claude перечитывает релевантную principles-страницу. Это часть фазы «подумай» в [CLAUDE.md](../CLAUDE.md).
