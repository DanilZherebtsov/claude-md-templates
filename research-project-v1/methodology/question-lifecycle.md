# Жизненный цикл исследовательских вопросов

Исследовательский вопрос — это формулировка того, что нужно выяснить: подвопрос главного исследовательского вопроса, методологический выбор, требующий разбора, или конкретное затыкание дыры в обзоре. Его жизнь — четыре стадии, по подпапкам в `output/questions/`:

```
output/questions/
  open/            ← Вопрос сформулирован, в активную работу не взят.
                    Аналог backlog'а — то, что хочется выяснить, но не сейчас.
  investigating/   ← В работе сейчас. Обычно один-два файла одновременно.
                    Содержит черновики ответа, выписки, заметки по ходу.
  answered/        ← Ответ найден. Ждёт извлечения claim'ов и/или synthesis-страницы.
  abandoned/       ← Отказались — вопрос оказался неверно сформулирован, или
                    исследование сменило фокус. Оставляем для истории.
```

**Имена файлов:** `q-<NNN>-<slug>.md` (например, `q-014-bias-samozvanca.md`). Номер сквозной, не сбрасывается между папками — при перемещении файл сохраняет своё имя.

## Структура файла вопроса

```markdown
# Q-014 — Полная формулировка вопроса

**Status:** open | investigating | answered | abandoned
**Date:** YYYY-MM-DD
**Связи:** ссылки на claim'ы, ADR, другие вопросы, которые этот вопрос затрагивает.

## Почему важно

<один абзац: зачем отвечать на этот вопрос, какое решение или утверждение от него зависит>

## Что нужно для ответа

<источники, методы, шаги. Может быть чек-листом>

## Гипотеза / предполагаемый ответ

<если есть рабочая версия — записать. Будет проверена по ходу>

## Заметки по ходу (для investigating)

## [YYYY-MM-DD] заголовок
<наблюдения, выписки, промежуточные выводы>

## Ответ (для answered)

<краткая формулировка ответа со ссылками на claim'ы, которые из него извлечены,
и на synthesis-страницу, если из ответа выросла длинная форма>
```

## Переходы между папками

1. **open → investigating** — человек или Claude по запросу человека переносит файл, когда вопрос берётся в работу. STATE.md обновляется одновременно. Обычно одновременно «в работе» один-два вопроса; больше — сигнал, что фокус размывается.
2. **investigating → answered** — после того как ответ сформулирован и подкреплён источниками. **При этом переходе Claude предлагает:**
   - Извлечь атомарные claim'ы в `wiki/claims/`. Один ответ часто рождает несколько claim'ов — каждый со своим evidence.
   - Если ответ объёмный (обзор, сравнение, многошаговое обоснование) — создать `wiki/synthesis/` страницу с длинной формой. Synthesis ссылается на извлечённые claim'ы.
   - Если в ответе появился методологический выбор — извлечь ADR в `wiki/decisions/`.
   Файл вопроса в `answered/` остаётся как полная история; claim'ы и synthesis — canonical-выжимка для будущих читателей. Двунаправленные ссылки: claim/synthesis в `sources:` указывает на файл вопроса, файл вопроса — на claim/synthesis.
3. **investigating → abandoned** или **open → abandoned** — отказались. Опционально создаётся ADR (если отказ — методологическое решение) или просто остаётся файл с пометкой причины. Полезно, если позже возникнет соблазн вернуться.
4. **answered → investigating** (повторное открытие) — допустимо, если найден новый источник, опровергающий ответ. В этом случае:
   - Файл вопроса возвращается в `investigating/`.
   - Связанные claim'ы пересматриваются (часть может стать `superseded` или `invalidated`).
   - Synthesis-страница обновляется или помечается `superseded` с указанием на новую версию.

## Правила

- Claude не перемещает файлы между папками автоматически. Только по явной просьбе («перенеси вопрос Q-014 в answered») или после подтверждения предложения («вопрос отвечен, перенести в answered и извлечь claim'ы?»).
- Вопрос в `open/` или `investigating/` живёт «свободно» — его можно дописывать, переписывать, разбивать. В `answered/` и `abandoned/` вопрос фактически immutable: содержательные пересмотры идут через возврат в `investigating/`, не правкой задним числом.
- **Не плодите вопросы спекулятивно.** Хороший research-вопрос либо что-то блокирует, либо появился из конкретного пробела в обзоре. Вопрос «а интересно было бы изучить X» без привязки к текущему фокусу — это шум, который засоряет backlog. Если фокус сменится — сформулируем заново.
- Если вопрос слишком крупный (требует нескольких независимых исследований) — разбейте на подвопросы с общим префиксом (`q-014a-...`, `q-014b-...`); они движутся по стадиям независимо.
