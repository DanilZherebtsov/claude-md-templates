# CLAUDE.md — LLM-вики для исследовательского проекта

Базовая идея: не выводить одно и то же знание из сырых источников заново при каждом вопросе. Вместо этого — однократно компилировать каждый источник в постоянную, взаимосвязанную вики. Дальше Claude читает вики, а не сырые источники, когда отвечает. К сырому возвращается только тогда, когда нужно интегрировать новый источник, проверить цитату или разрешить противоречие.

Корневой файл содержит **always-on правила и указатели**. Детальные процедуры (ingest, query, lint, конвенции страниц, lifecycle вопросов, bootstrap) — в [methodology/](methodology/), читаются по триггеру.

---

## О проекте

> Заполните один раз, обновляйте по мере значительных изменений. Здесь — **стабильный контекст исследования**, который не меняется от недели к неделе. Текущий фокус и открытые вопросы — в `STATE.md`. Claude читает этот раздел при каждом обращении.

**Главный исследовательский вопрос:** <одно предложение — что именно мы пытаемся выяснить>.

**Аудитория результата:** <кто потребитель — академическое сообщество, заказчик, мы сами, рынок>. Это определяет жанр deliverable (статья / отчёт / брифинг / датасет).

**Рамки и ограничения:** <методологические выборы, scope, временной горизонт, доступ к данным, этические рамки, дедлайн>. Подробности и эволюция — в `wiki/`.

## Архитектура

Структура проекта в корне. Центральные слои — `raw/` (входные источники) и `wiki/` (скомпилированные знания). Рядом — методология, рабочая поверхность, место под датасеты:

```
raw/             ← Источники. Вы добавляете, Claude только читает.
                   Подпапки — по типу источника. Стандартные:
  literature/    (статьи, книги, главы, препринты, технические отчёты —
                  всё, что прошло какую-то форму внешней публикации)
  fieldwork/     (расшифровки интервью, полевые заметки, наблюдения,
                  опросы — первичные данные, собранные нами)
  conversations/ (переписка с экспертами, разборы с коллегами, лекции,
                  записи рабочих обсуждений)
  decisions/     (заметки со встреч, голосовые, треды — то, что повлияло
                  на методологический выбор)
                   Опционально, под нужды проекта:
  datasets/      (описания и снапшоты используемых датасетов —
                  сами данные живут в data/, здесь — метаданные и контекст)
wiki/            ← Скомпилированные знания. Управляет Claude, поддерживается
                   автоматически.
                   Четыре type-папки, плоско (никаких подпапок внутри типа):
  claims/        (атомарные утверждения с цепочкой evidence; центральный тип —
                  основная единица накопленного знания. Группировка через
                  префиксы имени файла: topic-, hypothesis-, finding-, definition-)
  decisions/     (методологические ADR — выбор метода, scope, рамки,
                  трактовка термина; связаны через supersession)
  synthesis/     (записанные обратно сквозные ответы: обзоры, сравнения,
                  ретроспективы, брифинги, главы итогового deliverable)
  principles/    (исследовательские правила, выведенные из инцидентов
                  и повторяющихся ловушек; читаются перед нетривиальной задачей)
  index.md       (каталог — одна строка на страницу)
  log.md         (лог операций, только дополнение)
methodology/     ← Инструкции для Claude: процедуры операций и форматы.
                   Корневой CLAUDE.md содержит указатели; детали — здесь.
                   Часть шаблона, копируется вместе с CLAUDE.md, читается по триггеру.
  ingest.md             (процедура добавления знания)
  query.md              (процедура структурированного ответа)
  lint.md               (правила проверки целостности)
  page-conventions.md   (frontmatter, типы страниц, формат claim-evidence)
  question-lifecycle.md (поток вопроса: STATE → output → wiki)
  state-rules.md        (структура и правила обновления STATE.md)
  index-log-format.md   (формат index.md и log.md)
  bootstrap.md          (как развернуть структуру с нуля)
output/          ← Рабочие артефакты Claude: черновики, выписки, документы
                   для согласования. Не canonical, рабочий стол. Можно подчищать.
                   Здесь же — рабочий файл активного вопроса (`q-NNN-<slug>.md`:
                   заметки по ходу, черновик ответа). Когда ответ готов, он
                   уезжает в wiki/claims/ + wiki/synthesis/, а файл — расходный.
                   Повестка вопросов живёт не здесь, а в STATE.md.
  drafts/        (черновики финальных deliverable — статьи, отчёты, презентации)
data/            ← Опционально. Датасеты, на которых строится исследование:
                   сырые (immutable), обработанные, выходные. Появляется по
                   необходимости. Не в `wiki/` — это материал анализа, не знание.
                   Контракт (структура + происхождение + метод обработки)
                   описывается в `wiki/claims/` или `wiki/synthesis/`.
src/             ← Опционально. Скрипты анализа, симуляции, расчёты —
                   если исследование требует кода. Не появляется автоматически.
CLAUDE.md        ← Этот файл.
STATE.md         ← Операционное состояние: текущий фокус, открытые вопросы,
                   что в работе, блокеры. Не canonical (намерения, не факты).
```

Правила:

- **`raw/` неизменяем.** Только добавление. Никогда не редактировать и не удалять — оригинальная формулировка источника важна, когда мы потом перепроверяем цитату или утверждение. Если источник перестал быть актуальным, страница в `wiki/`, ссылающаяся на него, помечается `superseded`; сам файл в `raw/` остаётся.
- **`wiki/` под управлением Claude.** Человек читает, но не правит руками — все мутации wiki делает Claude. Растёт через несколько каналов: (а) ingest сырого источника из `raw/`; (б) запись-обратно содержательного ответа в `synthesis/` после query; (в) извлечение claim-ов и ADR из отвечённого вопроса; (г) запись принципа из инцидента или обсуждения. Если в вики что-то неверно — исправьте источник в `raw/` (или укажите Claude, где ошибка), и Claude перекомпилирует.
- **Глубина wiki = 1.** Внутри `wiki/` — только один уровень тематических подпапок (четыре типа выше), глубже не идти. Если в одном type-folder скапливается 30+ однородных страниц — это сигнал к **горизонтальному расширению** (новый top-level type или плоская группировка через префиксы имён), не к подпапкам. См. раздел «Дисциплина», правило про расширения схемы.
- **`raw/` — исключение.** Внутри `raw/` глубина по необходимости разрешена: это хранилище под навигацию человеком (например, `raw/literature/<автор>/`, `raw/fieldwork/<кейс>/<дата>/`).
- **`methodology/` — часть шаблона, не рабочая область.** Правки только человеком и только при пересмотре методологии исследовательского шаблона как такового. Расширение разрешено (новые файлы для новых операций); переписывание существующих требует осознанного решения.
- **Источники побеждают вики.** Если страница в `wiki/` говорит одно, а оригинальный источник в `raw/` — другое: истина в источнике, пока новый ingest или ADR не зафиксирует обратное. Вики — наш скомпилированный конспект, не первоисточник.
- **Каждое утверждение — с цитатой.** Любой claim или ключевой тезис в synthesis обязан иметь ссылку на конкретное место источника (файл + страница / параграф / таймкод). Это lint-проверяемо; без локализации цитата считается недокументированной.
- **Имена файлов.** Kebab-case, описательные, ≤ 60 символов. Сырые файлы префиксуются `YYYY-MM-DD-` (дата добавления в проект); оригинальная дата публикации идёт в metadata-заголовок самого файла.

## Иерархия источников

В одной сессии Claude видит несколько источников знаний. Чтобы не было путаницы, какой из них первичный, — порядок жёсткий:

1. **CLAUDE.md** (этот файл) — always-on правила и контекст проекта.
2. **`methodology/`** — детали операций и форматов. Читаются по триггеру, не в фоне.
3. **Вики** (`wiki/`) — canonical-знание, накопленное через ingest.
4. **Сырые источники** (`raw/`) — первоисточник, к которому возвращаемся при сомнениях.
5. **Auto-memory Claude** (`MEMORY.md`, `memory/`) — быстрый кэш между сессиями, **не canonical**.

**Конфликт между memory и вики.** Всегда верить вики, не memory. Memory может быть устаревшей; вика отражает последний ingest.

**Конфликт между вики и сырым источником.** При расхождении — перепроверить источник и исправить вики. Не предполагать, что вики «знает лучше» — именно так заводится дрейф.

**Запись в memory.** Когда появилось новое знание о проекте — Claude сначала кладёт его в `wiki/` через ingest. В memory, опционально, оставляет **короткий указатель**: тип записи + ссылка на страницу вики. Длинное содержание не дублировать.

**`STATE.md` — отдельная ось, не часть иерархии выше.** STATE описывает текущие намерения и прогресс исследования, а не утверждения о мире. С wiki не конфликтует (разные домены: wiki — что мы выяснили, STATE — что мы делаем).

## Операционное состояние

`STATE.md` в корне репозитория — единственное место, где живут текущий фокус и прогресс исследования. **Структура — фиксированный набор из 7 разделов** (стадия, путь до цели, в работе, следующее, отвечено, блокеры, открытые вопросы низкого приоритета). Пустые разделы остаются с пометкой `_пусто_`.

Детали — структура, правила обновления, триггеры свежести, связь с wiki — в [methodology/state-rules.md](methodology/state-rules.md). Прочитайте перед первой работой со STATE или при ревизии правил.

**Триггеры чтения STATE.md (для Claude):**
- При старте сессии — молча читать STATE.md. Это даёт контекст «на чём остановились» без явного вопроса.
- На вопросы «на чём остановились», «что в работе», «что дальше», «какие блокеры», «какие вопросы открыты» — STATE.md основной источник ответа.
- Если поле `_Обновлено:_` старше 7 дней — в первой реплике сессии предложить «STATE устарел на N дней, что изменилось?».

## Wiki: типы страниц и операции

Wiki содержит **4 типа страниц**:

- **`claims/`** — атомарные утверждения. Одна страница = одно утверждение + цепочка evidence (поддерживающие и противоречащие источники с локализованными цитатами) + связи с другими claim'ами (`supports`, `contradicts`, `refines`). Это центральный тип; всё знание исследования сходится сюда. Группировка через префиксы: `topic-`, `hypothesis-`, `finding-`, `definition-`.
- **`decisions/`** — методологические ADR. Применяются к методологическим выборам: какой метод, какие границы, какая трактовка термина, какой scope, что исключаем. Решения помечаются как `superseded`, а не удаляются.
- **`synthesis/`** — записанные обратно ответы: обзоры литературы, ретроспективы, сквозные сравнения, брифинги, главы итогового deliverable. Это «длинная форма» — несколько claim'ов, связанных в нарратив.
- **`principles/`** — исследовательские правила, выведенные из инцидентов и повторяющихся ловушек (например, «всегда проверять цитату по оригиналу», «не использовать confidence-числа»). Читаются Claude перед нетривиальной задачей.

Frontmatter, формат шаблонов по типам (включая claim-evidence формат), страницы-журналы, перекрёстные ссылки — [methodology/page-conventions.md](methodology/page-conventions.md).

**Три операции** работают с wiki. Запускаются обычной фразой в чате; Claude сам определяет по смыслу, какую вы имеете в виду.

- **Ingest** («обработай эту статью», «запомни это», «добавь в вики») → [methodology/ingest.md](methodology/ingest.md).
- **Query** («что мы знаем про X», «какие свидетельства за/против Y», «сделай обзор по теме Z») → [methodology/query.md](methodology/query.md).
- **Lint** («прогони lint», «проверь вики») → [methodology/lint.md](methodology/lint.md).

**Поток исследовательских вопросов** (повестка в STATE → рабочий файл в output → выжимка в wiki) — [methodology/question-lifecycle.md](methodology/question-lifecycle.md).

**Формат `index.md` и `log.md`** — [methodology/index-log-format.md](methodology/index-log-format.md).

## Дисциплина (то, что не даёт вики сгнить)

1. **Фильтровать на входе, а не на хранении.** Если источник не стоит интеграции — не интегрируй его. Не полагайся на последующую чистку. В вики попадает только то, что ты готов защищать.
2. **Supersession вместо тихого исчезновения.** Никакого автоматического забывания. Старый материал остаётся, помеченный как `superseded` и связанный с заменой. Утверждение, опровергнутое полгода назад, часто ценнее свежего — именно его вы вот-вот забудете и повторите.
3. **Никакой ложной точности.** Никаких числовых оценок уверенности. Достоверность claim'а видна из цепочки его evidence: количество источников, их жанр (рецензируемая статья vs пост в блоге), наличие contradicts-связей. «Три статьи и одно интервью» бьёт «0.85» каждый раз.
4. **Цитата с локализацией.** Любая ссылка на источник — с конкретным местом (страница, параграф, таймкод). Без локализации — это пересказ, а не цитата; lint помечает такие места.
5. **Человек в петле записи.** Claude предлагает; человек подтверждает любую нетривиальную мутацию вики. Хуки и автоматический ingest соблазнительны, но они молча портят вики. Медленнее — это нормально.
6. **Lint не опционален.** Прогон раз в неделю. **Claude отслеживает свежесть:** при старте сессии читает дату последней записи `## [YYYY-MM-DD] lint |` в `wiki/log.md`; если прошло > 7 дней — предлагает прогон в первой реплике. Если отчёты lint растут быстрее, чем вы успеваете их разбирать — пересматривать нужно схему, а не вики.
7. **Сначала схема, потом механизм.** Когда что-то ощущается неправильно — сначала поправьте этот файл или соответствующий файл в `methodology/`, а не пишите скрипты. Большинство «проблем тулинга» — это замаскированные проблемы схемы.
8. **Расширение схемы — только по горизонтали.** Допустимые расширения: новый top-level type в `wiki/` (плоско, рядом с четырьмя существующими) или новая top-level папка в корне (рядом с `raw/`, `wiki/`, `output/`, `methodology/`). **Углублять имеющиеся типы запрещено.** Никаких `wiki/claims/topic/`, `wiki/decisions/2026/`. Если в одном type-folder скапливается 30+ однородных страниц — расширяйте горизонтально либо группируйте через префиксы имени. `raw/` — исключение: там глубина по необходимости разрешена.

## Принципы работы Claude в задачах

Эти правила применяются ко всем задачам в проекте — анализ источников, формулировка вопросов, синтез, переписка. Они не про вики, а про то, как Claude действует.

Каждая нетривиальная задача проходит две фазы. Сначала **остановись и подумай**, потом действуй. Пропустить фазу «подумай» — самая частая причина того, что Claude уходит в ненужную работу или встраивает в неё ошибочные предпосылки.

**Нетривиальная задача** — та, где Claude должен **сделать выбор** (между трактовками, источниками, формулировками) или **построить план** (несколько шагов или несколько файлов). Конкретные триггеры: затрагивается методология / claim-цепочка / scope вопроса; нужно знание нескольких источников; есть риск исказить цитату или утверждение; непонятно, что точно нужно. **Тривиальные** (фаза «подумай» не нужна): опечатка, переименование по точной формулировке, чтение и пересказ одной страницы. **В сомнении — считать задачу нетривиальной.**

### Перед началом — фаза «подумай»

**Закрой все запросы в сообщении (даже для тривиальных задач).** Если просьб несколько — перечисли их списком в начале ответа и закрой каждую; конкретные action-items (команды, пути) давай первыми, до анализа. Не зарывайся в один под-вопрос, теряя остальные — повторяющаяся ошибка.

1. **Перечитай релевантные принципы.** Если задача методологическая — `wiki/principles/methodology.md`. Процессная — `wiki/principles/process.md`. Доменная — `wiki/principles/domain.md`. Принципы — это извлечённые из прошлых инцидентов правила; пропуск этого шага = повторение прошлых ошибок.
2. **Сформулируй предположения.** Назови вслух, что считаешь данным. Если есть неопределённость — спрашивай, не угадывай.
3. **Покажи разные трактовки.** Если запрос или источник можно понять двумя способами — обозначь оба, не выбирай молча.
4. **Не выдумывай.** Цифры, цитаты и факты — только из источников или вики, с указанием места (файл + страница / параграф). Если данных не хватает — `[требует уточнения]` вместо угаданного значения.
5. **Один вариант за раз.** Не предлагай десять трактовок сразу. Обсуждай по одной, начиная с самой консервативной.
6. **Возражай, когда оправдано.** Если запрос ведёт к перенатягиванию вывода — скажи об этом, а не выполняй молча.

### В процессе — фаза «действуй»

1. **Одна задача за раз.** Не перескакивать на соседние улучшения. Доводить текущее до конца, потом браться за следующее.
2. **Сначала простота.** Минимальная формулировка, которая решает задачу. Никаких спекулятивных оговорок, гибкости «на будущее», обработки невозможных контртезисов. Если получилось 200 строк там, где хватило бы 50 — переписать проще.
3. **Точечные изменения.** Менять только то, что необходимо. Не «улучшать» соседние страницы, формулировки, форматирование. Заметили несвязанную проблему — упомянуть, но не править.
4. **Выполнение от цели.** Для каждой нетривиальной задачи — сформулировать критерии успеха перед началом. «Разобраться в теме X» → «сформулировать 3 ключевых claim'а с источниками; перечислить открытые подвопросы».

### После — фиксация принципа, если он появился

Если из задачи родилось правило вида «всегда делать X / никогда Y» — Claude предлагает записать его в `wiki/principles/<applicability>.md` с источником (ссылка на инцидент в `raw/` или на текущее обсуждение). Принципы записываются только из конкретных случаев, не из общих соображений.

## Документы и нейминг

Касается **рабочих артефактов Claude** — черновиков, отчётов, экспортов. Артефакты внутри `wiki/` живут по правилам [methodology/page-conventions.md](methodology/page-conventions.md). Вопросы — по правилам [methodology/question-lifecycle.md](methodology/question-lifecycle.md).

- **Куда класть.** Всё, что Claude генерирует как deliverable, — в `output/`: черновики статей и отчётов в `output/drafts/`, рабочие файлы активных вопросов (`q-NNN-...`) — прямо в `output/`. Не canonical, можно подчищать. **Не в `wiki/`** — там только компилированное знание. **Не в `data/`** — там сами датасеты, не интерпретации.
- **Имена файлов.** На русском, через подчёркивание, с датой когда уместно: `обзор_литературы_2026-05-25.md`.
- **Даты.** В тексте — `ДД.ММ.ГГГГ`. В именах файлов и YAML — `YYYY-MM-DD`. Для источников из `raw/literature/` — префикс по дате добавления в проект; оригинальный год публикации — в метаданных файла.
- **Цитаты.** Всегда с локализацией: `(Smith 2023, p. 47)` или `[interview-acme-2026-05-12.md, мин. 14:30]`. Без локализации — `[требует уточнения локации]`.
- **Цифры в документах.** Всегда с источником. Без источника — пометка `[требует уточнения]` вместо угаданного значения.
