# Конвенции wiki-страниц — frontmatter, типы, формат

Прочитайте этот файл:
- перед созданием новой wiki-страницы;
- перед редактированием frontmatter;
- при ревизии правил оформления.

## Frontmatter

Каждая страница wiki начинается с YAML-заголовка:

```yaml
---
type: decision | architecture | discovery | synthesis | principles
status: active | superseded | draft
updated: YYYY-MM-DD
sources:                  # обязательно
  - ../raw/decisions/2026-05-18-auth-redesign-thread.md
  - https://datatracker.ietf.org/doc/html/rfc7636
implementation:           # обязательно для type: architecture
  - ../../src/auth/pkce.py
  - ../../src/auth/__tests__/pkce.test.py
supersedes: [adr-0007]    # опционально
superseded_by: [adr-0014] # опционально
tags: [auth, security]
---
```

**Поле `sources:` обязательно.** Утверждение без источника — это галлюцинация, замаскированная под факт. Lint следит за этим и не пропускает страницы без источников.

**Поле `implementation:` обязательно для `architecture`.** Это **навигация и path-check**, не drift-детектор. Lint проверяет, что все указанные пути существуют (битый путь = код был удалён или переименован, страница вики ссылается в пустоту). Семантическое расхождение «контракт в коде разошёлся с описанием на странице» lint не видит — это остаётся ответственностью человека (PR-review, периодический аудит). Поле может быть пустым на ранних черновиках страницы — тогда `status: draft`.

## Атомарность

Одна концепция на страницу. Мягкий лимит — 400 строк, жёсткий — 800. Если страница перевалила за 400 — выделите подстраницы и свяжите их. Ограниченный размер означает, что любое отдельное чтение остаётся в пределах контекстного окна.

## Перекрёстные ссылки

Везде markdown-ссылки с относительными путями:
- `[Название](../type/slug.md)` для wiki-страниц;
- `[Описание источника](../../raw/category/file.md)` для сырых файлов;
- `[Название](https://...)` для внешних URL.

Анкор-текст обязателен и человекочитаемый — не slug файла. Единый формат обеспечивает рендеринг в любом просмотрщике (включая GitHub) и согласованность с полем `sources:` в frontmatter.

## Страницы-журналы

Если страница накапливает наблюдения во времени (типично для `discovery/` или `synthesis/`), каждая запись внутри начинается с `## [YYYY-MM-DD] Заголовок`. Это даёт возможность читать, что свежо, а что устарело, без необходимости разбирать историю git.

## Типы страниц — формат

### decisions/ — Architecture Decision Records

Применяются как к продуктовым, так и к техническим выборам. Лёгкий формат ADR:

```
# ADR-0012 — Принять PKCE для нативного потока аутентификации

Status: active
Date: 2026-05-18
Context: <проблема и ограничения, вынудившие сделать выбор>
Decision: <что мы решили, одним предложением>
Consequences: <что становится проще, что — сложнее>
Alternatives considered: <кратко, с обоснованием отказа>
Sources: <цитаты>
```

Решения **помечаются как superseded, а не удаляются.** Старая запись остаётся читабельной, с заголовком, указывающим на её замену. Supersession лучше тихого исчезновения: устаревший ADR всё ещё объясняет, почему появился текущий.

### architecture/ — описание системы

Одна страница на модуль, сервис, интеграцию, контракт, инвариант, многошаговый workflow продукта или контракт runtime-данных (см. правило «Runtime-данные — в коде» в [CLAUDE.md](../CLAUDE.md)).

Зафиксируйте форму, зависимости, владельцев (внешний код помечается отдельно) и решения, которые этим управляют. Для workflow-процессов: вход, выход, шаги (нумерованно), правила/формулы внутри шагов.

Обязательная секция `## Implementation` со ссылками на конкретные файлы кода + дублирование в YAML-поле `implementation:`. Перекрёстно ссылайтесь на ADR-ы.

### discovery/ — данные о пользователях и рынке

Одна страница на сегмент пользователей, персону, job-to-be-done, гипотезу, конкурента, клиента или содержательный инсайт. У гипотез — явный статус (`open`, `validated`, `invalidated`) и доказательства, которые его сдвинули.

Группировка через префиксы имени файла, не подпапки: `client-acme.md`, `segment-smb.md`, `jtbd-cost-estimation.md`, `competitor-foo.md`. Это сохраняет инвариант «глубина = 1» и позволяет фильтровать через `ls client-*`.

### synthesis/ — записанные обратно ответы

Страницы, появившиеся из содержательных запросов: сравнения, ретроспективы, сквозные анализы, брифинги. Они накапливают результаты ваших исследований так же, как и источники, прошедшие ingest.

### principles/ — правила, выведенные из работы

Когда из инцидента, повторяющейся ошибки или удавшегося паттерна выводится правило вида «всегда делать X / никогда Y» — оно записывается сюда, в страницу по applicability (`process.md`, `architecture.md`, `domain.md` — или другое разбиение по необходимости).

Внутри страницы каждая запись:

```markdown
## P-XX — Краткое имя принципа

**Правило:** что делать / что не делать.

**Почему:** контекст, инцидент или соображение, из-за которого правило возникло.

**Когда применять:** триггер (тип задачи, контекст).

**Прецедент:** дата + один-два кейса, где правило сработало или было нарушено.

**Источник:** ссылка на инцидент в `raw/` или связанный ADR.
```

**Триггер записи (для Claude):** когда в разговоре или из инцидента появляется правило, которое будет применимо к будущим задачам того же типа, — Claude предлагает: «это похоже на принцип P-XX в `wiki/principles/<applicability>.md`, записать?». Принципы — не теория из головы, а извлечение из конкретного случая; без источника принцип не записывается.

**Триггер чтения (для Claude):** перед началом нетривиальной задачи Claude перечитывает релевантную principles-страницу (process для задач процесса, architecture для архитектурной правки, domain для доменной работы). Это часть фазы «подумай» из раздела «Принципы работы Claude в задачах» в [CLAUDE.md](../CLAUDE.md).
