# STATE.md — структура и правила обновления

`STATE.md` в корне репозитория — единственное место, где живут текущие планы и прогресс разработки. Wiki хранит знания (решения, факты, гипотезы); STATE хранит намерения (что в работе, что следующее, какие блокеры). У них разная природа: wiki — append-mostly с обязательными источниками и атомарностью; STATE — частые перезаписи без источников, всё в одном файле для обзорности.

## Структура

Файл — фиксированный набор из 7 разделов, всегда в одном порядке. Пустые разделы остаются с пометкой `_пусто_`, не удаляются: благодаря этому глаз привыкает к расположению и сразу видит изменения.

1. **Стадия** — где мы в lifecycle и какова текущая цель (идея / прототип / MVP / бета / v1 / v2).
2. **Путь до цели** — milestones от текущего момента до цели, по порядку. Весь путь в одном месте.
3. **Сейчас в работе** — что в флайте прямо сейчас (обычно одна-две задачи). Файл спеки — в `specs/` (`<feature>-NNN-...`, `status: active`).
4. **Следующее (1–2 недели)** — что подхватим после текущего. Бэклог; одностроки здесь, файл спеки в `specs/` заводится при взятии в работу.
5. **Завершено за последнюю неделю** — буферная зона перед схлопыванием.
6. **Блокеры и риски**.
7. **Известный техдолг** — список ссылок на ADR со статусом `active` и тегом `tech-debt`.

**Гранулярность отражает горизонт понимания.** В начале проекта «Путь до цели» — 2–3 строки крупными мазками («сформулировать концепцию», «10 интервью», «собрать MVP»). По мере приближения каждый пункт дробится на подпункты. Не пытайтесь атомизировать преждевременно — это создаёт ложную точность.

## Правила обновления

1. **Источник правды.** STATE — единственное место для текущего плана. Никаких параллельных списков в голове, заметках или чатах.
2. **Claude обновляет в трёх моментах:** (а) молча читает в начале сессии, упоминает только если состояние релевантно текущему запросу; (б) после завершения значимого пункта переносит его из «Сейчас в работе» в «Завершено за последнюю неделю»; (в) после ingest, который меняет приоритеты, — предлагает правку, человек подтверждает.
3. **Человек обновляет в двух:** когда смещает приоритеты и когда добавляет новый блокер. Можно через чат («обнови STATE, добавь блокер X»), можно руками.
4. **Свежесть по дате.** Поле `_Обновлено:_` в начале файла — обязательно. **Claude при старте сессии читает это поле; если дата старше 7 дней — предлагает в первой же реплике: «STATE устарел на N дней, что изменилось?»** Это правило, не скрипт; механизм может быть автоматизирован отдельно при желании (lint в CI, bash-хук), но базовый контракт — Claude-check.
5. **Связь с wiki через markdown-ссылки.** Любой пункт, ссылающийся на знание, делает это через `[Название](wiki/type/slug.md)`. Сами знания живут в wiki — в STATE только формулировка задачи и ссылка.
6. **«Завершено» — буферная зона.** Висит неделю, потом схлопывается: всё, что породило wiki-страницу, заменяется на markdown-ссылку на эту страницу; строчка «N задач закрыто за период YYYY-MM-DD ... YYYY-MM-DD, см. git log» заменяет рутинную работу без wiki-следа; остальное удаляется. Память о выполненной работе живёт в git и в ADR, не в STATE.
7. **STATE не canonical.** При конфликте с кодом или wiki — правда в коде/wiki. STATE отражает намерение, а не факт. Lint может пометить расхождение, но не правит автоматически.
