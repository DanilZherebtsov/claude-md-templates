# CLAUDE.md — LLM-вики для разработки программного продукта

Базовая идея: не выводить одно и то же знание из сырых документов заново при каждом вопросе. Вместо этого — однократно компилировать каждый источник в постоянную, взаимосвязанную вики. Дальше Claude читает вики, а не сырые источники, когда отвечает. К сырому возвращается только тогда, когда нужно интегрировать новые данные или разрешить противоречие.

Корневой файл содержит **always-on правила и указатели**. Детальные процедуры (ingest, query, lint, конвенции страниц, lifecycle спек, bootstrap) — в [methodology/](methodology/), читаются по триггеру.

---

## О проекте

> Заполните один раз, обновляйте по мере значительных изменений. Здесь — **стабильный контекст продукта**, который не меняется от недели к неделе. Текущая стадия и что в работе — в `STATE.md`. Claude читает этот раздел при каждом обращении.

<1–2 предложения: что строим и для кого>

**Ограничения, которые должны влиять на любое решение:** <технологические выборы, ценности, табу, бюджет, тайминг>. Подробности и эволюция — в `wiki/`.

## Архитектура

Структура проекта в корне. Центральные слои — `raw/` (входные материалы) и `wiki/` (скомпилированные знания). Рядом — методология, рабочая поверхность для артефактов и место под код продукта:

```
raw/             ← Сырые источники. Вы добавляете, Claude только читает.
                   Подпапки — по типу источника. Стандартные:
  discovery/     (расшифровки интервью, опросы, страницы конкурентов, исследования рынка)
                   Внутри — подпапки по клиенту/сегменту:
                   raw/discovery/<client-slug>/YYYY-MM-DD-interview.md
  decisions/     (заметки со встреч, голосовые, переписка — то, что повлияло на решение)
  technical/     (RFC, документация вендоров, разборы инцидентов, дизайн-референсы)
                   Опционально, под нужды стадии:
  business/      (бизнес-модель, юнит-экономика, монетизация, финансовые расчёты)
  brainstorms/   (вижн, гипотезы до валидации, неструктурированные идеи)
wiki/            ← Скомпилированные знания. Управляет Claude, поддерживается автоматически.
                   Пять type-папок, плоско (никаких подпапок внутри типа):
  decisions/     (ADR-ы — что мы решили и почему; связаны через supersession)
  architecture/  (модули, сервисы, контракты, инварианты, многошаговые workflow продукта)
  discovery/     (пользователи, сегменты, JTBD, гипотезы, конкуренты)
                   Группировка через префиксы имени файла, не подпапки:
                   client-acme.md, segment-smb.md, jtbd-cost.md, competitor-x.md
  synthesis/     (записанные обратно ответы и сквозные анализы)
  principles/    (правила, выведенные из инцидентов и повторяющихся паттернов;
                   читаются Claude перед нетривиальной задачей)
  index.md       (каталог — одна строка на страницу)
  log.md         (лог операций, только дополнение)
methodology/     ← Инструкции для Claude: процедуры операций и форматы.
                   Корневой CLAUDE.md содержит указатели; детали — здесь.
                   Часть шаблона, копируется вместе с CLAUDE.md, читается по триггеру.
  ingest.md            (процедура добавления знания)
  query.md             (процедура структурированного ответа)
  lint.md              (правила проверки целостности)
  page-conventions.md  (frontmatter, типы страниц, формат)
  spec-lifecycle.md    (pending/active/accepted/rejected)
  state-rules.md       (структура и правила обновления STATE.md)
  index-log-format.md  (формат index.md и log.md)
  bootstrap.md         (как развернуть структуру с нуля)
output/          ← Рабочие артефакты Claude: отчёты, экспорты, черновики, документы
                   для согласования. Не canonical, рабочий стол. Можно подчищать.
  drafts/        (черновики deliverable и документы для согласования)
specs/           ← Task-спеки разработки: проблема, план, критерии приёмки.
                   Плоско, один файл на задачу: `<feature>-NNN-<slug>.md`.
                   Статус — во frontmatter (active | completed | rejected),
                   НЕ в подпапках (иначе дублирует STATE.md). STATE.md — индекс
                   активных; specs/ — сами документы. Не расходный мусор (как
                   output/) и не код: версионируется, completed/rejected хранятся
                   как история. Бэклог задач — в STATE.md, файл заводится при
                   взятии в работу. Веха из ≥3 задач → спринт-спека
                   `SPRINT-<NAME>.md` (`kind: sprint`): список задач живёт
                   в ней, не в STATE; мутабельна пока активна.
src/             ← Исходный код продукта (появится с началом разработки).
                   Финальный источник правды о поведении продукта — здесь, не в вики.
data/            ← Runtime-данные продукта: промпты для LLM, KB-словари, конфиги,
                   шаблоны email и т.п. Появляется по необходимости. Не в `wiki/` —
                   их потребляет рантайм, не Claude. Контракт (схема + способ
                   обновления) описывается в `wiki/architecture/`.
CLAUDE.md        ← Этот файл.
STATE.md         ← Операционное состояние: где мы в разработке, что в работе,
                   что следующее. Не canonical (отражает намерения, не факты).
```

Правила:

- **`raw/` неизменяем.** Только добавление. Никогда не редактировать и не удалять — оригинальная формулировка важна, когда мы потом перепроверяем утверждения.
- **`wiki/` под управлением Claude.** Человек читает, но не правит руками — все мутации wiki делает Claude. Растёт через несколько каналов: (а) ingest сырого источника из `raw/`; (б) запись-обратно содержательного ответа в `synthesis/` после query; (в) извлечение ADR из принятой спеки; (г) запись принципа из инцидента или разговора. Брейнштормы и решения, возникшие в чате, попадают в вики тем же путём: Claude сохраняет запись разговора в `raw/brainstorms/` или `raw/decisions/`, дальше — ingest. Если в вики что-то неверно — исправьте источник в `raw/` (или укажите Claude, где ошибка), и Claude перекомпилирует.
- **Глубина wiki = 1.** Внутри `wiki/` — только один уровень тематических подпапок (пять типов выше), глубже не идти. Если в одном type-folder скапливается 30+ однородных страниц — это сигнал к **горизонтальному расширению** (новый top-level type или плоская группировка через префиксы имён), не к подпапкам. См. раздел «Дисциплина», правило про расширения схемы.
- **`raw/` — исключение.** Внутри `raw/` глубина по необходимости разрешена: это хранилище под навигацию человеком, а не структура знания.
- **`methodology/` — часть шаблона, не рабочая область.** Правки только человеком и только при пересмотре методологии. Расширение разрешено (новые файлы для новых операций); переписывание существующих требует осознанного решения.
- **Код побеждает вики.** Когда появится `src/`: если `wiki/architecture/` говорит одно, а код — другое, истина в коде, пока ADR явно не зафиксирует обратное. Вики описывает систему концептуально, не подменяет код.
- **`specs/` — рабочие документы задач, не код и не вики.** Спека мутабельна, пока `status: active`; после `completed`/`rejected` замораживается (правки — через новый ADR или новую спеку, не задним числом). Статус — во frontmatter, не в подпапках. `completed`-спека — снимок намерения на момент разработки, не источник правды о текущем поведении (правду см. в коде и ADR). Детали — [methodology/spec-lifecycle.md](methodology/spec-lifecycle.md).
- **Runtime-данные — в `data/`, не в `wiki/`.** Промпты, KB-словари, конфиги-файлы, шаблоны email живут в top-level папке `data/`. В `wiki/architecture/` лежит **контракт**: схема данных, способ обновления, ссылки на код через поле `implementation:`.
- **Структура растёт по мере надобности.** Не подошёл источник под существующие подпапки `raw/` — заведи новую (в `raw/` глубина свободна; в `wiki/` — плоско, глубина = 1). Изменил раскладку — отрази её в дереве этого файла и в затронутых контекстных файлах.
- **Имена файлов.** Kebab-case, описательные, ≤ 60 символов. Сырые файлы префиксуются `YYYY-MM-DD-`, если известна дата источника.

## Иерархия источников

В одной сессии Claude видит несколько источников знаний. Чтобы не было путаницы, какой из них первичный, — порядок жёсткий:

1. **CLAUDE.md** (этот файл) — always-on правила и контекст проекта.
2. **`methodology/`** — детали операций и форматов. Читаются по триггеру, не в фоне.
3. **Вики** (`wiki/`) — canonical-знание, накопленное через ingest.
4. **Сырые источники** (`raw/`) — первоисточник, к которому возвращаемся при сомнениях.
5. **Auto-memory Claude** (`MEMORY.md`, `memory/`) — быстрый кэш между сессиями, **не canonical**.

**Конфликт между memory и вики.** Всегда верить вики, не memory. Memory может быть устаревшей; вика отражает последний ingest.

**Запись в memory.** Когда появилось новое знание о проекте — Claude сначала кладёт его в `wiki/` через ingest. В memory, опционально, оставляет **короткий указатель**: тип записи + ссылка на страницу вики. Длинное содержание не дублировать.

**`STATE.md` — отдельная ось, не часть иерархии выше.** STATE описывает текущие намерения и прогресс разработки, а не утверждения о мире. С wiki не конфликтует (разные домены: wiki — что мы знаем, STATE — что мы делаем). При конфликте STATE с кодом или wiki-страницей о фактах — правда в коде/wiki.

## Операционное состояние

`STATE.md` в корне репозитория — единственное место, где живут текущие планы и прогресс разработки. **Структура — фиксированный набор из 7 разделов** (стадия, путь до цели, в работе, следующее, завершено, блокеры, техдолг). Пустые разделы остаются с пометкой `_пусто_`.

Детали — структура, правила обновления, триггеры свежести, связь с wiki — в [methodology/state-rules.md](methodology/state-rules.md). Прочитайте перед первой работой со STATE или при ревизии правил.

**Триггеры чтения STATE.md (для Claude):**
- При старте сессии — молча читать STATE.md. Это даёт контекст «на чём остановились» без явного вопроса.
- На вопросы «на чём остановились», «что в работе», «что дальше», «какие блокеры», «какой техдолг» — STATE.md основной источник ответа.
- Если поле `_Обновлено:_` старше 7 дней — в первой реплике сессии предложить «STATE устарел на N дней, что изменилось?».

**Все стартовые проверки молчаливы.** Свежесть STATE и свежесть lint (§5 «Дисциплины») упоминаются в первой реплике **только при отклонении**. Сказать нечего — Claude молчит, «всё в порядке» не перечисляет.

## Wiki: типы страниц и операции

Wiki содержит **5 типов страниц**:

- **`decisions/`** — Architecture Decision Records. Применяются к продуктовым и техническим выборам. Решения помечаются как `superseded`, а не удаляются.
- **`architecture/`** — описание системы: модули, сервисы, интеграции, контракты, инварианты, многошаговые workflow продукта. Обязательная секция `## Implementation` со ссылками на код.
- **`discovery/`** — данные о пользователях и рынке: сегменты, персоны, JTBD, гипотезы, конкуренты, клиенты, инсайты. У гипотез — явный статус (`open`, `validated`, `invalidated`).
- **`synthesis/`** — записанные обратно ответы: сравнения, ретроспективы, сквозные анализы, брифинги.
- **`principles/`** — правила, выведенные из инцидентов и повторяющихся паттернов. Читаются Claude перед нетривиальной задачей.

Frontmatter, формат шаблонов по типам, страницы-журналы, перекрёстные ссылки — [methodology/page-conventions.md](methodology/page-conventions.md).

**Три операции** работают с wiki. Запускаются обычной фразой в чате; Claude сам определяет по смыслу, какую вы имеете в виду.

- **Ingest** («обработай этот документ», «запомни это», «добавь в вики») → [methodology/ingest.md](methodology/ingest.md).
- **Query** («что мы знаем про X», «сделай справку по Y», «сравни A и B») → [methodology/query.md](methodology/query.md).
- **Lint** («прогони lint», «проверь вики») → [methodology/lint.md](methodology/lint.md).

**Поток task-спек** (бэклог в STATE → файл в `specs/` со `status: active` → извлечение ADR при acceptance, спека замораживается в `completed`) — [methodology/spec-lifecycle.md](methodology/spec-lifecycle.md).

**Спринты.** Веха, требующая **≥3 task-спек**, оформляется спринт-спекой (`SPRINT-<NAME>.md`, `kind: sprint`): цель, список задач со статусами, acceptance, журнал; мутабельна пока активна. По ходу: задача → task-спека → реализация → проверка → (при необходимости правка спринт-спеки) → следующая → закрытие спринта. **Список задач живёт в спринт-спеке, не в STATE** (STATE — индекс активного спринта и активной задачи). Одиночная задача спринта не требует. Детали — [methodology/spec-lifecycle.md](methodology/spec-lifecycle.md).

**Формат `index.md` и `log.md`** — [methodology/index-log-format.md](methodology/index-log-format.md).

## Дисциплина (то, что не даёт вики сгнить)

1. **Фильтровать на входе, а не на хранении.** Если источник не стоит интеграции — не интегрируй его. Не полагайся на последующую чистку. В вики попадает только то, что ты готов защищать.
2. **Supersession вместо тихого исчезновения.** Никакого автоматического забывания. Старый материал остаётся, помеченный как `superseded` и связанный с заменой. Баг, зафиксированный полгода назад, часто ценнее свежего — именно его вы вот-вот забудете.
3. **Никакой ложной точности.** Никаких числовых оценок уверенности. Достоверность утверждения видна из цепочки его источников, связанных решений и подтверждающих коммитов. «Два интервью и один ADR» бьёт «0.85» каждый раз.
4. **Человек в петле записи.** Claude предлагает; человек подтверждает любую нетривиальную мутацию вики. Хуки и автоматический ingest соблазнительны, но они молча портят вики. Медленнее — это нормально.
5. **Lint не опционален.** Прогон раз в неделю. **Claude отслеживает свежесть:** при старте сессии читает дату последней записи `## [YYYY-MM-DD] lint |` в `wiki/log.md`; если прошло > 7 дней — предлагает прогон в первой реплике («прошло N дней с последнего lint, прогнать?»). Если отчёты lint растут быстрее, чем вы успеваете их разбирать — пересматривать нужно схему, а не вики.
6. **Сначала схема, потом механизм.** Когда что-то ощущается неправильно — сначала поправьте этот файл или соответствующий файл в `methodology/`, а не пишите скрипты. Большинство «проблем тулинга» — это замаскированные проблемы схемы.
7. **Расширение схемы — только по горизонтали.** Допустимые расширения: новый top-level type в `wiki/` (плоско, рядом с существующими пятью) или новая top-level папка в корне (рядом с `raw/`, `wiki/`, `output/`, `methodology/`). **Углублять имеющиеся типы запрещено.** Никаких `wiki/discovery/clients/`, `wiki/decisions/2026/`, `wiki/architecture/modules/auth/`. Если в одном type-folder скапливается 30+ однородных страниц — расширяйте горизонтально либо группируйте через префиксы имени. `raw/` — исключение: там глубина по необходимости разрешена.
8. **Runtime-данные — в `data/`, не в `wiki/`.** Промпты, KB-словари, конфиг-файлы, шаблоны email лежат в top-level папке `data/`. Их потребляет рантайм продукта; Claude не читает их как знание. В `wiki/architecture/` лежит **контракт** (схема + способ обновления), связанный с кодом через поле `implementation:`. Положить runtime-файл в `wiki/` — типовая ошибка, которая медленно заваливает вики мусором.

## Принципы работы Claude в задачах

Эти правила применяются ко всем задачам в проекте — кодинг, дискавери, документы, переписка. Они не про вики, а про то, как Claude действует.

Каждая нетривиальная задача проходит две фазы. Сначала **остановись и подумай**, потом действуй. Пропустить фазу «подумай» — самая частая причина того, что Claude уходит в ненужную работу или встраивает в неё ошибочные предпосылки.

**Нетривиальная задача** — та, где Claude должен **сделать выбор** (между подходами, файлами, формулировками) или **построить план** (несколько шагов или несколько файлов). Конкретные триггеры: затрагивается архитектура / контракты / runtime-данные; нужно знание нескольких файлов; есть риск сломать существующее поведение; непонятно, что точно нужно. **Тривиальные** (фаза «подумай» не нужна): опечатка, переименование по точной формулировке, чтение и объяснение кода, ответ из одной wiki-страницы. **В сомнении — считать задачу нетривиальной.**

### Перед началом — фаза «подумай»

**Закрой все запросы в сообщении (даже для тривиальных задач).** Если просьб несколько — перечисли их списком в начале ответа и закрой каждую; конкретные action-items (команды, пути) давай первыми, до анализа. Не зарывайся в один под-вопрос, теряя остальные — повторяющаяся ошибка.

1. **Перечитай релевантные принципы.** Если задача архитектурная — `wiki/principles/architecture.md`. Процессная — `wiki/principles/process.md`. Доменная — `wiki/principles/domain.md`. Принципы — это извлечённые из прошлых инцидентов правила; пропуск этого шага = повторение прошлых ошибок.
2. **Сформулируй предположения.** Назови вслух, что считаешь данным. Если есть неопределённость — спрашивай, не угадывай.
3. **Покажи разные трактовки.** Если запрос можно понять двумя способами — обозначь оба, не выбирай молча.
4. **Не выдумывай.** Цифры и факты — только из исходников или вики, с указанием источника. Если данных не хватает — `[требует уточнения]` вместо угаданного значения.
5. **Один вариант за раз.** Не предлагай десять решений сразу. Обсуждай по одному, начиная с самого простого.
6. **Возражай, когда оправдано.** Если запрос ведёт к плохому решению — скажи об этом, а не выполняй молча.

### В процессе — фаза «действуй»

1. **Одна задача за раз.** Не перескакивать на соседние улучшения. Доводить текущее до конца, потом браться за следующее.
2. **Сначала простота.** Минимальный код или текст, который решает задачу. Никаких спекулятивных абстракций, гибкости «на будущее», обработки невозможных сценариев. Если получилось 200 строк там, где хватило бы 50 — переписать проще.
3. **Точечные изменения.** Менять только то, что необходимо. Не «улучшать» соседний код, форматирование, комментарии. Заметили несвязанный мёртвый код — упомянуть, но не удалять.
4. **Выполнение от цели.** Для каждой нетривиальной задачи — сформулировать критерии успеха перед началом. «Исправить баг» → «написать тест, воспроизводящий баг; добиться его прохождения».

### После — фиксация принципа, если он появился

Если из задачи родилось правило вида «всегда делать X / никогда Y» — Claude предлагает записать его в `wiki/principles/<applicability>.md` с источником (ссылка на инцидент в `raw/` или на текущую дискуссию). Принципы записываются только из конкретных случаев, не из общих соображений.

## Документы и нейминг

Касается **рабочих артефактов Claude** — отчётов, экспортов, документов для согласования. Артефакты внутри `wiki/` живут по правилам [methodology/page-conventions.md](methodology/page-conventions.md). Спеки задач — по правилам [methodology/spec-lifecycle.md](methodology/spec-lifecycle.md).

- **Куда класть.** Всё, что Claude генерирует как deliverable, — в `output/`: черновики, отчёты, экспорты, документы для согласования. Не canonical, можно подчищать. **Не в `wiki/`** — там только LLM-генерируемые компилированные знания. **Не в `src/`** — там код продукта. Когда у продукта появится shipping-документация (README, гайды, API docs), она пойдёт в отдельную папку `docs/` рядом с `src/`.
- **Имена файлов.** На русском, через подчёркивание, с датой когда уместно: `сравнение_подходов_2026-05-25.md`.
- **Даты.** В тексте — `ДД.ММ.ГГГГ`. В именах файлов и YAML — `YYYY-MM-DD`.
- **Денежные суммы.** Рубли с пробелом-разделителем тысяч: `220 000 ₽`. Другие валюты — по соглашению самой валюты (`$1,200`, `€990`).
- **Цифры в документах.** Всегда с источником (файл, строка или ссылка). Без источника — пометка `[требует уточнения]` вместо угаданного значения.
