# STATE.md — структура и правила обновления

`STATE.md` в корне репозитория — единственное место, где живут текущий фокус и прогресс исследования. Wiki хранит знание (claim'ы, методологические решения, синтезы, принципы); STATE хранит намерения (что в работе, что следующее, какие блокеры, какие вопросы открыты). У них разная природа: wiki — append-mostly с обязательными источниками и атомарностью; STATE — частые перезаписи без источников, всё в одном файле для обзорности.

## Структура

Файл — фиксированный набор из 7 разделов, всегда в одном порядке. Пустые разделы остаются с пометкой `_пусто_`, не удаляются: благодаря этому глаз привыкает к расположению и сразу видит изменения.

1. **Стадия** — где мы в lifecycle и какова текущая цель (разведка / систематический обзор / сбор данных / анализ / синтез / оформление).
2. **Путь до цели** — milestones от текущего момента до цели, по порядку. Весь путь в одном месте.
3. **Сейчас в работе** — какие вопросы в флайте прямо сейчас (обычно один-два). Рабочий файл вопроса — в `output/` (`q-NNN-...`).
4. **Следующее (1–2 недели)** — что подхватим после текущего. Повестка; одностроки здесь, файла нет, пока вопрос не взят в работу.
5. **Отвечено за последнюю неделю** — буферная зона перед схлопыванием.
6. **Блокеры и риски**.
7. **Открытые вопросы низкого приоритета** — backlog без срока, одностроки прямо здесь (без файлов).

**Гранулярность отражает горизонт понимания.** В начале проекта «Путь до цели» — 2–3 строки крупными мазками («разметить поле», «8 интервью», «черновик обзора»). По мере приближения каждый пункт дробится на подпункты. Не пытайтесь атомизировать преждевременно — это создаёт ложную точность.

## Правила обновления

1. **Источник правды.** STATE — единственное место для текущего фокуса. Никаких параллельных списков в голове, заметках или чатах.
2. **Claude обновляет в трёх моментах:** (а) молча читает в начале сессии, упоминает только если состояние релевантно текущему запросу; (б) после того, как вопрос отвечён и из него извлечены claim/synthesis, переносит его из «Сейчас в работе» в «Отвечено за последнюю неделю»; (в) после ingest, который меняет приоритеты или открывает новые подвопросы, — предлагает правку, человек подтверждает.
3. **Человек обновляет в двух:** когда смещает фокус и когда добавляет новый блокер. Можно через чат («обнови STATE, добавь блокер X»), можно руками.
4. **Свежесть по дате.** Поле `_Обновлено:_` в начале файла — обязательно. Стартовый триггер свежести (что делать, если устарело) живёт в CLAUDE.md (always-on); здесь не дублируется.
5. **Связь с wiki через markdown-ссылки.** Любой пункт, ссылающийся на знание, делает это через `[Название](wiki/<type>/<slug>.md)`. Сами claim'ы, ADR, synthesis живут в wiki — в STATE только формулировка и ссылка.
6. **«Отвечено» — буферная зона.** Висит неделю, потом схлопывается: всё, что породило claim или synthesis, заменяется на ссылку; строчка «N вопросов закрыто за период YYYY-MM-DD ... YYYY-MM-DD, см. git log» заменяет рутинные мелкие закрытия; остальное удаляется. Память об отвечённом живёт в git и в claims/synthesis, не в STATE.
7. **STATE не canonical.** При конфликте с вики или сырыми источниками — правда в вики/raw. STATE отражает намерение, а не утверждение. Lint может пометить расхождение, но не правит автоматически.
