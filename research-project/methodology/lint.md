# Lint — проверка здоровья вики

Запускается фразами: «прогони lint», «проверь вики», «проверь целостность». Стартовый триггер свежести (когда сам предложить прогон) — в CLAUDE.md, §6 «Дисциплины».

Периодическая проверка. Два уровня с разной степенью полномочий.

## Автоисправление (детерминированное — Claude правит сам)

- Страницы, отсутствующие в `wiki/index.md` → добавить с заглушкой `(no summary)`.
- Записи индекса, указывающие на удалённые файлы → пометить `[MISSING]` (не удалять; решение оставить за человеком).
- Битые внутренние ссылки, для которых есть ровно один файл с тем же именем в другом месте → починить путь.
- Ссылки в поле `sources:`, указывающие на перемещённые сырые файлы с единственным кандидатом → починить.
- Битые claim-связи (`supports / contradicts / refines`), указывающие на переименованный claim с единственным кандидатом → починить.
- Отсутствующие взаимные ссылки между очевидно связанными страницами в одной папке → добавить (например, claim A с `supports: [B]`, но в B нет упоминания A → добавить упоминание в раздел «Связи» страницы B).

## Только отчёт (эвристическое — требует суждения человека)

- Страницы-сироты без входящих ссылок.
- Концепции, упомянутые на нескольких страницах, но без собственной claim-страницы.
- Устаревшие claim'ы, противоречащие более новому источнику.
- Claim'ы со статусом `active`/`validated`, чьи Counter-evidence заметно перевешивают Evidence — кандидаты на ревизию.
- ADR-ы со статусом `active`, чьё решение противоречит более новому ADR.
- Страницы больше 800 строк (и claim'ы больше 100 строк — кандидаты на разбиение через `refines`).
- Страницы без поля `sources:`.
- **Цитаты без локализации:** упоминание автора/источника без указания страницы / параграфа / таймкода. Lint выдаёт список таких мест в формате `<страница>:<номер строки> — <фрагмент>`.
- **Нарушение глубины:** любой `.md` в `wiki/<type>/<subdir>/...` — страницы вики должны быть плоско в type-folder (см. правило про расширения схемы в «Дисциплине» в [CLAUDE.md](../CLAUDE.md)). Исключение — `wiki/index.md` и `wiki/log.md` на уровне `wiki/`.

## Проверка обновлений шаблона

Этот проект инстанцирован из версионируемого шаблона (имя и версия — в строке-метке вверху `CLAUDE.md`). На каждом прогоне lint:

1. **Гейт по версии.** Прочитать апстрим-`CHANGELOG.md`: `https://raw.githubusercontent.com/DanilZherebtsov/claude-md-templates/main/<имя-шаблона>/CHANGELOG.md` (`<имя-шаблона>` — из строки-метки). Верхняя версия = версии проекта → ничего не выводить. Недоступен → сказать, не молчать.
2. **Апстрим новее — сверять САМИ ФАЙЛЫ, не только changelog.** Changelog бывает скупым и прячет правки за фразой вроде «выровнен по base». Поэтому скачать файлы шаблона из зеркала (`git clone https://github.com/DanilZherebtsov/claude-md-templates` во временную папку, либо raw-файлы `CLAUDE.md` + все `methodology/*.md`) и **сравнить со СТАНДАРТ-частями этого проекта** (каркасы `methodology/` + base-owned проза `CLAUDE.md`, вне регионов `<<СЛОТ …>>`).
3. **Оценить радиус каждого различия.** *Инструкция* (правка `methodology/` или прозы `CLAUDE.md`; в changelog без метки или `[инструкция]`) — применяется обновлением файла, существующие знания не трогаются. *Формат файлов знаний* (новое поле/раздел, переименование типа; в changelog помечено `[формат]`) — требует **миграции** существующих страниц `wiki/`.
4. **Предложить по реальному диффу.** Инструкционные — применить точечно, с подтверждения; слоты и доменные разделы не трогать. Для `[формат]` — назвать цену (сколько страниц затронет) и предложить стратегию: **lazy** (мигрировать при следующем касании, по умолчанию) / **eager** (все сразу) / **skip** (не принимать — легитимно, проект вправе разойтись). **Файлы знаний массово не переписывать без явного согласия.** После применения — обновить строку-метку версии.

После lint — дописать в `wiki/log.md`:
```
## [YYYY-MM-DD] lint | найдено N проблем, исправлено M
```
