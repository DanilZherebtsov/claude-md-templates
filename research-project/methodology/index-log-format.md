# Формат `index.md` и `log.md`

## index.md

Группируется по типу (четыре секции: `claims`, `decisions`, `synthesis`, `principles`), внутри типа — по алфавиту. Каждая строка:

```
- [Название страницы](<type>/<slug>.md) — однострочное резюме. _Updated: YYYY-MM-DD_
```

Для claim'ов резюме — это сама формулировка claim'а в одно предложение, плюс статус в скобках:
```
- [claim-attention-bandwidth](claims/claim-attention-bandwidth.md) — рабочая память ограничена ~4 элементами в активном поддержании _(validated)_. _Updated: 2026-05-25_
```

Ссылка указывается **относительно самого `index.md`**, который лежит в `wiki/`. Префикс `wiki/` не нужен — иначе при просмотре получится `wiki/wiki/<type>/<slug>.md`. То же правило действует и для ссылок между страницами вики: путь относителен от текущей страницы (например, со страницы `wiki/claims/foo.md` на `wiki/synthesis/bar.md` — это `../synthesis/bar.md`, а не `wiki/synthesis/bar.md`).

## log.md

Только дополнение, новые записи внизу. Префикс записи `## [YYYY-MM-DD] <op> | <subject>` делает лог удобным для grep: команда `grep "^## \[" wiki/log.md | tail -20` показывает свежую активность.

Дата записи — **дата окружения на момент записи**, не mtime файла и не дата, когда делалась сама работа. Из режима «только дополнение» следует инвариант: дата каждой новой записи **≥** даты предыдущей. Свежая запись с более ранней датой почти всегда значит, что дата взята с mtime, — это нарушение, которое ловит lint.

Операции, попадающие в лог:
- `ingest | <название источника>` — после каждого ingest, с перечнем каскадно-обновлённых страниц.
- `lint | найдено N проблем, исправлено M` — после каждого lint.
- `bootstrap | инициализация` — единожды, при первом разворачивании структуры.
- `question-closed | Q-NNN` — когда вопрос отвечён и из него извлечены claim/synthesis. Запись содержит ссылки на извлечённые страницы.
- (опционально) другие явные операции, которые человек хочет иметь в истории.
